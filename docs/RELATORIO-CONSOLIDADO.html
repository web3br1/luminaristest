§1 · Qualidade de código universal

  • tsc limpo é gate, não meta — server e my-app sem novos erros.
  • Zero any evitável — interfaces locais ou unknown + narrowing. Exceção única: Record<string, any> dos dados dinâmicos. catch (e) com narrowing, nunca catch (e: any).
  • Imports reais (nunca inventados) · nada de segredos (via env).
  • Convenção de nomes — PascalCase / use<X> / I<X>.
  • Sem lógica morta / TODO falso · logger.error(msg, { context }) (string primeiro).

§2 · Arquitetura backend

Route → Controller → Service → Repository → Prisma
                        ↘ Policy
  • Separação inviolável: Route (só roteia) · Controller (Zod+actor+service+handleApiError, zero prisma/regra) · Service (regra+policy-first, zero prisma direto/Express) · Repository (único com prisma.*) · Policy (só boolean, zero throw/dados).
  • DI via Factory (nunca new Repository()) · Policy-first antes de dados.
  • Erros tipados de lib/errors · cross-tenant = NotFoundError (anti-enumeration).
  • Soft-delete universal (finds com deletedAt: null; nunca .delete()) · findAll com $transaction([findMany, count]).
  • Validação Zod no controller · actor IUser | null · tipos de 'generated/prisma'.
  • Resposta { success, data } · registro de rota = 3 toques (index + protectedApiPaths + OpenAPI — senão 401 silencioso).
  • Domínios DynamicTable — orquestra DynamicTableService, resolve por internalName.
  • Money — addMoney(); previousValue = count>0 ? total/count : undefined; single-pass · escritas de chat retornam { status: 'PROPOSED' }.

§3 · Arquitetura frontend

  • Pages Router + auth guard + getServerSideProps/serverSideTranslations.
  • Service layer (nunca fetch direto; tipos locais; zero any).
  • Reuse canônicos (§0) · detalhe = MODAL, não rota.
  • Paginação ao ler DynamicTable (API máx. 50; fetch-all até totalPages; validar >50).
  • useMemo([deps]) em dados derivados · container full-height (sem max-w-* divergente).
  • Dynamic imports p/ libs pesadas · SSR-safe · views agrupadas filtram pelo pai ativo · resolver por internalName · i18n sem hardcode.

§4 · Design system

  • neutral-*, NUNCA zinc-* (grep deve ser vazio); borda dark = dark:border-neutral-800.
  • Cards = rounded-2xl/3xl; inputs/botões = rounded-xl. Wrappers de libs/listas/timeline também são cards.
  • Tipografia — font-semibold corpo / font-black ênfase; labels text-[10px] uppercase tracking-widest.
  • Badges color/10+color/20+color-600 · paleta blue/emerald/rose/amber.

§5 · Testes

  • beforeEach(() => jest.clearAllMocks()) · floats com toBeCloseTo(v, 2).
  • referenceDate fixo · cross-tenant NotFoundError · buildService(overrides?) · KPI Empty Safety (0, nunca NaN).

§6 · Hierarquia de verificação

tsc < build/SSR (200) < hidratação (use PROD) < interatividade
  • Telas atrás de withAuth → build de produção; cor/raio por estilos computados (on-brand = rgb(23,23,23); zinc-900 = rgb(24,24,27) off-brand); rodar testes da feature.
Anti-omissão: antes de "pronto", confirme que nenhum item aplicável ficou sem resposta. Skill omissa → o contrato prevalece.
