Guia das Skills do Luminaris

Como usar o sistema de geração de código do monorepo: 34 skills geradoras organizadas por camada, orquestradas por 3 agentes, governadas por um auditor que impede drift. Este guia é o ponto de entrada — os contratos detalhados vivem nos arquivos referenciados ao longo do texto.

34/34 skills validated baseline skills-governance-v1 gate de CI: skill-audit run --all

1 · Visão geral — o modelo mental

O Luminaris é um monorepo com dois lados: server/ (Express + Prisma, camadas estritas) e my-app/ (Next.js Pages Router). As skills são instruções reutilizáveis que geram código consistente com os padrões do repo — cada uma ensina o contrato de uma camada e aponta para o exemplar canônico que deve ser espelhado.

Três ideias sustentam tudo:

  • Reuse antes de recriar. Antes de gerar "X novo", verifica-se se o canônico já existe (GenericTable, Modal, StandardPagination, AnalyticsDashboard…). Bespoke só com divergência sancionada de shape ou posse (critério em .claude/skills/_REUSE-CRITERION.md).
  • Camadas não são over-engineering. A cadeia Route → Controller → Service → Repository → Prisma (+ Policy), injeção via Factory, DTO Zod, soft-delete e o registro de rota em 2 toques são requisitos — não se "enxuga" uma policy ou um DTO.
  • O contrato é o gate final. Toda skill herda .claude/skills/_ARCHITECTURE-CONTRACT.md; em conflito, o contrato prevalece.
Fontes da verdade. skill define (o que gerar) · governance relaciona (regra→gate→eval) · cbm localiza (codebase-memory, índice estrutural — não-autoritativo) · código/testes confirmam (verdade objetiva) · skill-audit impede drift (gate determinístico no CI).

2 · STOP — onde o módulo/feature vive

Antes de qualquer linha, responda a pergunta binária. Ela decide a tecnologia e todo o resto do roteamento já assume essa decisão tomada.

O dado/módulo é…TecnologiaSkill
Tabela que o usuário cria/configura em runtime (CRM, formulários, fluxos adaptáveis)DynamicTable (preset + plugin)dynamic-table-preset-generator
Entidade com invariante financeiro/legal/regulatório que o banco deve garantir (contábil, folha, fiscal, RH)Prisma first-class (Model + Service + Repo + Policy)fullstack-feature-generator
Em dúvida → Prisma first-class. "ERP" sozinho não decide. E nunca injete um serviço Prisma first-class (PostingService, PayrollService…) dentro de DynamicTableService, RuleContext ou um RulePlugin. Integração entre os dois mundos (ex.: venda→lançamento contábil) sobe a um controller/serviço de integração — nunca dentro do motor de plugins. Regra completa: _ARCHITECTURE-CONTRACT.md §2.1.

3 · Como invocar uma skill

Há três caminhos, do mais guiado ao mais direto:

a) Pelo orquestrador (recomendado para tarefas reais)

Descreva a tarefa em linguagem natural e deixe o luminaris-orchestrator escolher as skills certas, na ordem certa, e delegar. Ele roda o STEP 0 (§2.1) e monta um plano com passos, dependências, checks e riscos.

# o agente analisa, classifica e produz o PLANO DE EXECUÇÃO
/luminaris-orchestrator "quero um módulo de folha de pagamento do zero"

b) Direto, uma skill específica

Quando você já sabe exatamente qual camada quer gerar:

/frontend-modal-generator "modal de confirmação de exclusão de Lead"
/backend-service-generator "InvoiceService com create/getById/update/delete"

Cada skill aceita um argument-hint próprio (nome do recurso, ação, modo) — veja o catálogo abaixo.

c) Auditoria (manutenção, não geração)

/skill-audit governance-check   # coerência regra↔gate↔eval + self-check do auditor
/loop /skill-audit --sweep      # varredura comportamental completa (human-run)

4 · O loop de 3 agentes — separação estrita de papéis

Tarefas não-triviais passam por três agentes com fronteiras rígidas. Cada um faz uma coisa e delega o resto — nenhum fecha o loop sozinho.

orchestrator

Decompõe & roteia

Analisa a tarefa, roda o gate §2.1, escolhe skills + ordem, monta o plano. Não implementa nem aprova. Nunca se auto-atribui implementação/revisão.

→
implementer

Edita & verifica

Executa o plano lendo cada skill + refs; roda os checks de fato (tsc/jest) com exit codes reais. Para no vermelho. Não se auto-revisa nem mexe em governança.

→
reviewer

Verifica & reporta

Valida contra os padrões de camada e a fronteira §2.1; reporta PASS/FAIL com arquivo:linha. Evidência ausente → BLOCKED, nunca PASS. Devolve o defeito — não conserta em silêncio.

O handoff carrega contexto. Escopo da tarefa · arquivos alterados · checks executados + exit codes · riscos não resolvidos · localização das evidências. É isso que mantém os papéis verificáveis em vez de "confio no anterior".

Garantias gateadas (cobertas por evals comportamentais):

  • Implementer afirma PASS sem rodar os checks → reviewer rejeita.
  • Falta evidência → reviewer devolve BLOCKED, nunca PASS.
  • Reviewer acha defeito → devolve ao implementer (não edita o código sob revisão).
  • Orchestrator não atribui implementação/revisão a si mesmo.
  • Implementer não altera governança/status de aprovação sem ser tasked explicitamente.

7 · Governança & skill-audit

Cada skill governada tem um governance.md que mapeia regra → gate → eval, e o auditor skill-audit garante a coerência dessa malha. O baseline está congelado em skills-governance-v1 e protegido por um gate de CI.

Comandos do auditor

node .claude/skills/skill-audit/skill-audit.mjs <comando>

inventory          # matriz de cobertura → governance/INVENTORY.md
validate           # estrutura + frontmatter + nomes + ids + referências
governance-check   # malha regra↔gate↔target↔eval
sync-metadata      # eval-score do SKILL.md == projeção do REPORT.md
controls           # prova que cada assertion corrigida ainda discrimina
self-check         # audita o auditor contra fixtures
run                # suíte completa; exit≠0 se qualquer finding

Gate de CI (obrigatório)

O job skill-governance em .github/workflows/ci.yml roda skill-audit run --all em todo PR e na main. É um required status check com strict: true — merge bloqueado se o gate ficar vermelho. É determinístico (estático, < 1s); a varredura comportamental (re-geração via subagente) continua human-run.

Para mudar uma regra ([PREFIX-NNN]), no mesmo change-set: o gate (governance.md), a eval (evals.json), o control quando a assertion foi de-brittled (controls/), o REPORT.md atualizado e o bump de governance-version. O eval-score é sempre projeção do REPORT — nunca editado à mão. E: nenhuma expansão do harness sem uma falha real reproduzível (YAGNI).

Como uma eval é provada

  1. Um subagente gera código em contexto limpo (lê só o SKILL.md, não as assertions).
  2. A saída bruta é salva verbatim em reports/<skill>/_eval.out.txt (sem trim manual), com marcadores ===<caso>=== e // caminho/arquivo.
  3. batch-eval extrai cada seção mecanicamente e roda as assertions (contains/regex/absent-code/AST), file-scoped por @arquivo::.
  4. Toda assertion corrigida ganha um control discriminante (variante boa passa, variante violadora falha).
