Bugs encontrados e corrigidos
1
cashflowKpis — Contas a Pagar sempre zero
Arquivo: SalesModule.ts · params.expenseDateField
Corrigido

O processor do cashflow usava expenseDateField: 'paymentDate' para filtrar despesas. Despesas pendentes não têm paymentDate (ainda não foram pagas), então eram silenciosamente ignoradas. Resultado: Contas a Pagar Total, Contas a Pagar Vencidas, Índice de Liquidez e Solvência = 0.

antes
expenseDateField: 'paymentDate'
depois
expenseDateField: 'date'
2
profitKpis — 5 KPIs sempre zero (funcionários, equity, horas)
Arquivo: SalesModule.ts · params estáticos ausentes
Corrigido

O processor profitKpis calcula Lucro por Funcionário, Lucro por Hora Trabalhada, Rentabilidade Geral (%), Produtividade do Lucro e Índice de Qualidade do Lucro a partir de params estáticos (employeesCount, totalHoursWorked, equity, recurringProfit). Esses params não estavam definidos no preset → todos defaultavam para 0.

antes
params: { period: 'month', monthsWindow: 12, ... // sem employeesCount/equity/horas }
depois
params: { ..., employeesCount: 3, totalHoursWorked: 528, equity: 50000, recurringProfit: 3500, }
3
revenueByType — "Não Operacional" sempre zero
Seed: revenueType com valor errado
Corrigido

O schema define o campo select com opção 'NonOperational' (sem hífen). O seed usava 'Non-Operational' (com hífen). O labelMap do chart mapeia NonOperational → 'Não Operacional', então a dimensão com hífen não era renomeada e acumulava receita no label errado, deixando "Não Operacional" = 0.

antes (seed)
revenueType: 'Non-Operational'
depois (seed)
revenueType: 'NonOperational'
4
costKpis — Variável, Impostos e Não Recorrente sempre zero
Seed: datas futuras dentro do mês corrente excluídas pela janela
Corrigido

O processor usa datePreset: 'thisMonth' cujo currentEnd = hoje (não fim do mês). As despesas variáveis/impostos/não-recorrentes do mês corrente estavam no dia 13–22, portanto no futuro em relação a hoje (12/06). Sendo excluídas da janela, seus totais no mês corrente ficavam zerados.

antes (seed)
paymentDate: monthDate(0, 13) // 13/jun → futuro paymentDate: monthDate(0, 22) // imposto → futuro
depois (seed)
paymentDate: monthDate(0, 4) // 04/jun → passado paymentDate: monthDate(0, 3) // 03/jun → passado
5
productsProfitMonthly e serviceEstimatedCost — zeros nos últimos 3 meses
Seed: saleItems populados apenas para os primeiros 60 registros
Corrigido

O seed original usava saleIds.slice(0, 60) para popular os itens de venda, cobrindo apenas até fevereiro/2026. Meses recentes (abr, mai, jun) não tinham itens, zeranando os KPIs de lucratividade por produto e custo estimado por serviço. Adicionalmente, responsibleEmployeeId estava ausente, tornando o custo estimado = 0 pois a fórmula depende do salário do funcionário vinculado.

antes (seed)
for (const {id: saleId} of saleIds.slice(0, 60)) { // sem responsibleEmployeeId await create(T.saleItems, { saleId, ... }); }
depois (seed)
for (const {id: saleId} of saleIds) { // todos await create(T.saleItems, { saleId, responsibleEmployeeId: emps[random], ... }); }

Conclusão

✓Sistema de Analytics validado

Todos os 26 endpoints de analytics do preset beautySalon respondem corretamente com dados significativos após o seed de teste. O único grupo com zeros é sales.revenueByChannel nos canais Telefone, App, Loja física e Outros — comportamento esperado e correto, pois esses canais simplesmente não existem nos dados do salão (apenas Direct, Online e Referral são usados).

Os 5 bugs corrigidos revelam padrões importantes a considerar na implementação de novos KPIs: (1) processar pendências requer o campo de data de criação, não de pagamento; (2) KPIs que dependem de parâmetros de negócio estáticos precisam de defaults no preset; (3) os valores de campos select devem ser idênticos entre o schema, o labelMap e os dados; (4) a janela thisMonth termina hoje, não no fim do mês.

Processadores validados
revenueKpis, profitKpis, cashflowKpis, costKpis, productCostKpis, temporalAggregation, aggregatePipeline, formulaCalculation, multiTableCalculation
Joins cross-tabela OK
sales→expenses (profit, cashflow), saleItems→sales→products→stockMovements (WAC), saleItems→services→employees (custo estimado)
Pontos de atenção
currentEnd = hoje (não fim do mês); pendências precisam do campo date; params estáticos do preset precisam de defaults de negócio documentados
