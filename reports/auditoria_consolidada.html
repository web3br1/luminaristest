Stack técnica & Higiene de dependências

Versões resolvidas nos lockfiles (não as declaradas no manifest) e avaliação de segurança/manutenção de cada pacote relevante.

🔴 jsonwebtoken 8.5.1 — vulnerável (alta confiança)

O lock resolve jsonwebtoken@8.5.1, anterior aos advisories de dez/2022 corrigidos em 9.0.0: CVE-2022-23539 (algoritmos irrestritos no verify), CVE-2022-23540 (chave none), CVE-2022-23541 (bypass de validação).

Diretamente relevante: jwt.verify(token, JWT_SECRET) sem allowlist de algorithms (jwt.ts:19) + fallback de segredo hardcoded 'your-jwt-secret-key' (jwt.ts:4). Os três fatores combinados = risco de forja de tokens admin. Ação: ≥9.0.0 + {algorithms:['HS256']} + exigir JWT_SECRET no boot — ou consolidar no jose (instalado, código morto).

§6 · Sistema

Módulos & Funcionalidades

Cada módulo com status real verificado no código — implementado, frágil, quebrado ou órfão.

Autenticação e Usuários implementada

Fluxo: POST /auth/register|login → bcrypt → JWT {id,username,role} (7d). authMiddleware protege paths por prefixo e injeta headers x-user-*.

  • logout é no-op — não invalida token nem limpa cookie.
  • register não valida o body (authController.ts:9).
  • Token em cookie não-httpOnly (exposto a XSS); jose é código morto incompatível (userId vs id).

Dynamic Tables — núcleo do ERP implementada

DynamicTableService.ts (988 linhas). Fluxo create: policy → valida Zod (runtime, contra schema da tabela) → regras avançadas (unique/compositeUnique/requiredIf/compare/noOverlap) → runRules(before) → persiste → runRules(after).

  • Governança declarativa real: deleteConstraints, immutableAfter, lifecycle, compositeUnique, noOverlap.
  • 🔴 isSystem vem do cliente (R2); compositeUnique faz full-scan O(n); findRowsReferencingId trunca em LIMIT 100 (delete-constraint pode furar); campos json são z.any().

Rules Engine frágil

10 plugins (SalesPlugin, StockMovementsApply, ProductAutoStock, UnitAutoStock, Commissions, Appointments, Employees, Goals, Leads, LeadsSeedOnUnit) despachados por RuleRegistry.ts.

🔴 Crítico (R1)

Sem transação — falha parcial corrompe dados. processSaleStockUpdate sem guarda de idempotência (double-apply em retry). getApplicable engole erros de supports() — regra crítica some silenciosamente. validateServiceDuration é no-op (AppointmentsPlugin.ts:65-70).

Presets de negócio implementada

POST /dashboard/create → merge Core + suite → installPresetAsSystem (3 passes: tabelas sem relações → resolve @@PRESET_TABLE_KEY:: → variante de saleItems). Install não transacional (ERP meio-instalado em falha); dois mecanismos paralelos de lookup (PresetService vs PresetManager) podem divergir.

Analytics / KPIs a parte mais madura

Registry pattern → processor por KPI → ChartDataPoint[] com drill-down. Única área com testes (6 arquivos: revenue, profit, cost, cashflow, sales, KpiEngine).

  • Sem cache — re-processa tudo a cada request; só RevenueKpiProcessor usa streaming.
  • 🟠 DataSanitizer: bug de locale US — "1,500" → 1.5 (DataSanitizer.ts:35).
  • 🟠 discoverKPIsAsync quebrado — measure sem type → throw no Compiler (AnalyticsService.ts:442).

Documentos + RAG implementada

Upload (multer memória, sem limite) → extractText síncrono → Document PENDING → background: chunks (500 palavras, overlap 50) → embedding por chunk → upsert Qdrant em lotes de 10.

  • ✅ Positivo: deleteDocument limpa Qdrant → chunks → SQL corretamente (DocumentService.ts:139-155), embora não-transacional.
  • 🔴 Busca vetorial sem filtro userId (R3); searchDocuments retorna chunkText: undefined (lê payload.text em vez de textContent, DocumentService.ts:174); documentos podem ficar presos em PROCESSING (sem watchdog).

Chat — RAG + Agente ERP implementada c/ gate

Tools read: list_my_tables, get_table_schema, query_table_data. Write (gera ActionProposal): request_record_creation/update. Gate: cliente reenvia confirmedProposalId → executeProposal checa proposal.userId === user.userId → executa via DynamicTableService (re-valida + policy + rules).

  • Modo selecionado pela presença de documentIds (não pelo ChatInstance.type).
  • __isSystem pode passar pela data da proposal; propostas nunca expiram (status EXECUTED/EXPIRED são código morto); KnowledgeGraph fica stale após delete de tabelas; custo OpenAI ilimitado.

Onboarding por IA (Interview) quebrado

⚠️ Library-only + endpoint inexistente

Backend sem controller/rota (admitido no próprio README). Frontend chama POST /dashboard/ai/ChatInterview que não existe. Bugs adicionais: regex JSON quebrada (/\\{[^\\}]*\\}/ — barras literais, nunca casa), uso errado de getChatCompletion(prompt, 'gpt-4-turbo') (o "modelo" vira system prompt), StateManager em memória. O onboarding funcional é o determinístico TotalControlSetup → POST /dashboard/create.

Outras

  • Dashboard layout: upsert por usuário (POST que atualiza — 201 enganoso). ✅
  • structuredData: backend completo, zero consumo no front — feature órfã (ver §12).
  • Reports: SSE com function-calling para gráficos sobre documentos. ✅
  • Chat instances/messages: CRUD ✅ (com vazamento de listagem — R6 — e forja de role — R24).

§10 · Sistema

Fluxos ponta a ponta

Os quatro fluxos mais importantes do sistema, com pontos de falha mapeados.

🛒 Criar uma venda (o mais crítico)

Usuário preenche SalesCreateModal → useSalesWizard valida (unidade, cliente, ≥1 item, preço>0).
FinanceService.createSaleWithItems → POST /dynamic-tables/:salesTableId/data (cria sale) → Promise.all criando cada item.
Backend: cada createTableData → valida schema → runRules.
SalesPlugin.beforeCreate (item): valida produto×serviço, checa estoque/reserva, auto-cria appointment p/ serviço.
Ao finalizar (status→Finalized): SalesPlugin.afterUpdate → baixa estoque, gera stockMovements, materializa commissions, atualiza métricas do customer.
🔴 Não atômico ponta a ponta (front E back)

Falha no item N não faz rollback dos anteriores nem da sale — front exibe N+1 toasts de sucesso; back fica em estado inconsistente. Estado alterado: sales, saleItems, productUnits, stockMovements, commissions, customers, appointments.

🤖 Chat com agente ERP → ação no banco

POST /api/chat (sem documentIds) → modo Agente → injeta AGENT_SYSTEM_PROMPT + dump do KnowledgeGraph.
Loop de até 5 tool-calls (gpt-4o): query_table_data (lê) ou request_record_creation/update (cria ActionProposal PENDING).
Resposta type:'ACTION_PROPOSAL' → front abre CommandConfirmationModal.
Usuário confirma → executeProposal (checa tenant ✅) → DynamicTableService (re-valida + policy + rules ✅) → deleta proposal.
⚠️ Riscos

__isSystem pode passar pela data da proposal; KnowledgeGraph stale (agente "vê" tabelas deletadas); prompt injection via conteúdo de query_table_data (mitigado pelo gate humano); custo OpenAI ilimitado.

📄 Upload e RAG de documento

POST /documents/upload (multer memória, sem limite) → extractText síncrono → Document PENDING → processamento em background (chunks → embeddings → Qdrant).
POST /api/chat com documentIds → embed da pergunta → vectorRepository.search(emb, 10, documentIds) → contexto → gpt-3.5-turbo.
🔴 R3 — cross-tenant

search não filtra userId → passar documentId de outro usuário vaza o conteúdo.

Correlações — achados que só aparecem cruzando as áreas

1 · A falta de transação é sistêmica, não pontual

Aparece no Rules Engine (RE-5, RE-11), no install de presets (PR-1) e na finalização de vendas. Causa raiz única: IDynamicTableRepository.ts não expõe primitiva de transação. Uma correção arquitetural (unit-of-work no repositório) resolve as três famílias de risco.

2 · LIMIT 100 implícito é um bug latente compartilhado

findRowsByFieldValue limita a 100 resultados e afeta 3 plugins: SalesPlugin/UnitAutoStock (estoque não provisionado), StockMovements (match de unitId errado) e LeadsPlugin (validação de transição de stage). Qualquer tenant que ultrapasse 100 registros nessas relações ativa o bug silenciosamente.

3 · O bypass __isSystem (R2) é maior do que parecia

Plugins usam ctx.isSystem para pular validações (appointments no passado, mix de tipos de item) e o schema usa para pular readOnly/immutableAfter/lifecycle (PR-7, DynamicTableService.ts:393-398). Como o R2 mostrou que __isSystem é controlável pelo cliente, todas essas validações são contornáveis por um usuário malicioso.

4 · Prompt injection compõe com o design schema-driven

O usuário controla nomes de tabelas/campos/opções (é a proposta do produto) e esses valores entram crus no system prompt do agente que tem tools de escrita (KnowledgeGraphService.ts:124-130). O caminho usuário → schema → prompt → tool de escrita existe de ponta a ponta; o modal de confirmação é a última linha de defesa.

5 · Funcionalidades existentes nas duas pontas, mas desconectadas

O backend calcula série histórica de 24 meses (fullRecords no RevenueKpiProcessor) mas o front renderiza sparkline com Math.random (FE-11). Análogo ao structuredData: pipeline backend 100% implementado, 0% consumido (DOC-4).
