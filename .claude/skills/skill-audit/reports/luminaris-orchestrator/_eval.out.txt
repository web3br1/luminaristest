## PLANO DE EXECUÇÃO — Módulo de Folha de Pagamento (Payroll)

**Tarefa:** "criar um módulo de Folha de Pagamento (payroll) do zero"

#### STEP 0 — Gate §2.1 (onde o módulo vive?) — executado ANTES de qualquer roteamento

| Pergunta | Resposta | Decisão |
|---|---|---|
| O usuário cria/configura o esquema em runtime? | NÃO — folha de pagamento tem estrutura fixa de negócio (proventos, descontos, base de cálculo, líquido), não definida pelo usuário em runtime | — |
| Tem invariante financeiro/legal/regulatório que o banco deve garantir? | SIM — folha é cálculo financeiro com obrigação legal/trabalhista; o líquido = proventos − descontos é invariante que o banco deve garantir, com unicidade (um demonstrativo por funcionário/competência) e atomicidade | **Prisma first-class** |
| A integridade depende de `@@unique`/FK/tipos reais? | SIM — `@@unique(funcionário, competência)`, FK para funcionário, valores em centavos (inteiro) | **Prisma first-class** |

**Veredito do STEP 0: Prisma first-class.** "Folha/RH" é categoria explicitamente listada no §2.1 como invariante legal/regulatório → roteia para `fullstack-feature-generator` (Model + Service + Repo + Policy próprios), NUNCA `dynamic-table-preset-generator`. Em dúvida o default já seria Prisma; aqui não há dúvida. Regra fixa: a futura integração folha→lançamento contábil (PostingService) NÃO entra neste plano como edição de `DynamicTableService`/plugin — sobe a um passo de controller/serviço de integração e fica fora deste escopo de criação do módulo.

**Risco:** High (fullstack-feature-generator + novo model Prisma com invariante financeiro)
**Branch recomendada:** `feature/payroll`

### Escopo da tarefa

Criar o módulo Payroll do zero como vertical-slice Prisma first-class, espelhando o exemplar `server/src/features/users/` e o módulo de referência `accounting` (Prisma first-class com tenancy + `@@unique` fechando TOCTOU). Backend completo (Prisma model → DTO → Repository → Policy → Service → Controller → Route + factory + registro de rota em 3 toques) com cobertura de testes do Service. Frontend NÃO está no escopo desta criação inicial salvo confirmação do usuário (ver Riscos).
